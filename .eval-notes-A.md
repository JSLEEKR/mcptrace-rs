## Security probes

- **Plaintext secret leak probe** — `{"api_key":"sk-REAL-SECRET-123","password":"hunter2-pw"}` fed through `proxy stdio` with jsonl + stdout exporter. Verdict: CLEAN. `grep -i "sk-REAL\|hunter2\|api_key\|password" captured.jsonl` → no matches. The span line only contains `arg_digest=0cfdb802...` (sha256) and `arg_bytes=56`. The stdout capture from the subprocess shows the child's own echo, which is normal transparent-proxy behavior (the proxy forwards wire bytes intact — that's the contract).
- **Oversized frame probe** — unit test `jsonrpc_frame_size_cap_enforced` exercises 2KB body with 1KB cap → `Error::FrameTooLarge` before parse. No OOM. README documents "oversized frames are dropped (not forwarded)" — this is a documented transparency break for DoS protection.
- **Deep nesting / long strings** — serde_json has its own recursion cap; size cap means a 1 MB string still parses, which is fine given we hash and drop the raw Value afterward.
- **Arg digest determinism** — hashes raw wire bytes (by design, not canonicalized). Doc in digest.rs explicitly states this tradeoff.

## Bugs found

### BUG-1 (CRITICAL, CORRECTNESS): SLO `error_rate` burn math was wrong

`evaluate` in `src/slo.rs` computed `budget = (1.0 - slo.target).max(1e-12)`
for the `error_rate` metric, i.e. it interpreted `target` as a target
*availability* even though the metric is named `error_rate` and the
README documents `target = 0.01  # <= 1% of calls can error` (i.e.
target is the *maximum allowed error rate*).

Impact: with the README's own example (`target = 0.01`, budget
should be 0.01), a 33% observed error rate produced
`burn_rate = 0.33 / 0.99 = 0.337` (silent, "ok"). After the fix it
produces `burn_rate = 0.33 / 0.01 = 33.3` (fires the alert). The old
code silently hid SLO violations — a showstopper for an SLO tool.

Edge cases:
- `target = 0` (zero tolerance): old code treated it as "100% error
  budget" so a single error was fine; new code treats it as zero
  tolerance and any error blows through.
- `target = 1` (100% errors allowed): old code gave enormous burn for
  any error at all; new code correctly returns low burn.
- Empty window: returns 0.0 burn in both old and new code (regression
  test added to prevent future regressions).

Fix location: `src/slo.rs` `evaluate()` (ErrorRate branch) + regression
tests + doc-comment rewrite. README burn-rate section rewritten to
match the three per-metric formulas. Availability math untouched
(it's correctly using the SRE-workbook formula since availability
really is `target = min uptime`).

Regression tests added:
- `slo::tests::evaluate_error_rate_within_budget_new_math`
- `slo::tests::evaluate_error_rate_zero_tolerance`
- `slo::tests::evaluate_error_rate_empty_window_not_burning`

Existing tests adjusted (their broken baselines depended on the old
math): `slo::tests::evaluate_error_rate_burning` (target 0.9 → 0.05)
and `tests/integration.rs::full_workflow_slo_check_from_jsonl`
(target 0.9 → 0.05). Both still exercise the burning path, now with
a sensible target.

## Non-bugs investigated but rejected

- **Digest not canonicalized across JSON key order**: deliberate per
  `src/digest.rs` doc — "hashing canonical wire bytes means digests
  are stable across mcptrace versions regardless of serde internals".
  Tests explicitly verify whitespace sensitivity. This is a design
  choice, not a bug.
- **`read_jsonl` halts on bad line instead of skipping**: strict-by-default
  matches what a forensic observability tool wants. Not a bug; not
  contradicted by README. Test `read_jsonl_surfaces_bad_line` confirms
  the design.
- **Line-based `BufReader::lines()` on tokio stdin/stdout**: reads as
  `String`, so non-UTF-8 bytes from a child would error the line read
  and break that direction. MCP JSON-RPC is ASCII-safe in practice,
  and the MCP spec itself uses UTF-8 JSON, so this is acceptable.
- **OTLP JSON `traceId`/`spanId` hex vs base64**: OTLP/JSON spec
  permits hex encoding of byte fields (the proto3 JSON mapping uses
  base64 for `bytes`, but modern collectors accept the hex form
  produced by the Go OTel JSON exporter). Common practice, not a bug.
- **Oversized frame silently dropped (not forwarded)**: transparency
  break but explicitly documented in README security section
  ("oversized frames are dropped (not forwarded)"). Trade-off the
  user signed up for.
