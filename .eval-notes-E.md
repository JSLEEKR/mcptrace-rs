## Probes run in cycle E (fresh angles)

### Timestamp precision — Span::now_unix_nanos
After D removed the `time` crate, `start_unix_nanos` is computed by
`Span::now_unix_nanos()` (span.rs:172-176):

    SystemTime::now()
        .duration_since(UNIX_EPOCH)
        .map_or(0, |d| d.as_nanos())

`Duration::as_nanos()` returns `u128` directly — no truncation, no
downcast. Stored field `start_unix_nanos: u128` (span.rs:78). The
documented "survive past 2554 CE" claim holds because we never coerce
to u64. On systems where the wall clock is pre-1970, the result is `0`
via the `map_or` (rather than panicking). Acceptable. ✓

### Clock monotonicity for duration measurement
proxy.rs:188 uses `started: Instant::now()` for the elapsed-time
measurement (proxy.rs:110: `p.started.elapsed().as_millis()`), and
only uses `SystemTime::now()` for the wall-clock `start_unix_nanos`
field. So even if the system clock jumps backwards mid-request, the
span `duration_ms` is still computed from the monotonic `Instant`,
not from `(end_unix_nanos - start_unix_nanos)`. Subtle but correct
— a real production-grade choice. ✓

### Unix nanos overflow at write time
`Span::end_unix_nanos` (span.rs:165-168) uses `saturating_add`. Test
`end_unix_nanos_saturates` pins behavior with `u128::MAX - 100` start
and `u64::MAX` duration. ✓

OTLP `render_span` (exporter.rs:236-237) uses raw `+`:
    let end_ns = start_ns + u128::from(s.duration_ms) * 1_000_000;
For any realistic `start_unix_nanos` (current epoch ≈ 1.78e18, well
under u128::MAX ≈ 3.4e38) plus any plausible `duration_ms`, this
cannot overflow. The crate-wide `#![deny(warnings)]` doesn't catch
it because it's not a warning, but in practice this is unreachable.
Not worth changing for v1.

### Zipkin/OTLP timestamp format correctness
- Zipkin v2 expects `timestamp` in **microseconds** since epoch and
  `duration` in microseconds. exporter.rs:120-121 computes
  `ts_micros = s.start_unix_nanos / 1_000` (nanos→micros, correct
  divisor 1000) and `dur_micros = u128::from(s.duration_ms) * 1_000`
  (millis→micros, correct multiplier 1000). Test
  `zipkin_timestamp_and_duration_micros` pins exact values
  `1_700_000_000_000_000` and `25_000` for a 25ms span at 1.7e18 ns.
  Math correct, off-by-1000x bug not present. ✓
- OTLP expects `startTimeUnixNano`/`endTimeUnixNano` as **strings**
  containing nanosecond integers. exporter.rs:262-263 uses
  `start_ns.to_string()` and `end_ns.to_string()` — strings, not
  numbers. Test `otlp_start_end_time_strings` pins exact string
  `"1700000000000000000"` and `"1700000000025000000"`. Spec compliant. ✓

### trace_id collision — splitmix64 seed
trace_id.rs:26-30: `seed()` returns
`SystemTime::now().duration_since(UNIX_EPOCH).map_or(0, |d| d.as_nanos() as u64)`.
The `as u64` truncates u128→u64 on the low bits, which for current
nanos epoch is fully populated entropy. `next()` then mixes in a
monotonic counter via `wrapping_add(c)` and runs splitmix64.

Within a single proxy session, two `next()` calls from the same
nanosecond differ only by the counter; splitmix64 is bijective so
they map to distinct outputs. trace_ids combine two `next()` calls
(`hi`, `lo`) for 128 bits of state. Test
`trace_ids_are_unique_across_1k_calls` confirms 1000 calls → 1000
unique ids. The doc honestly says "not cryptographic". For an
observability tool, fine. ✓

### arg_digest determinism — BTreeMap key order
jsonrpc.rs:109-115 calls `serde_json::to_vec(args)` where `args` is a
`&serde_json::Value`. serde_json's `Value::Object` is a `Map<String,
Value>`; with the default feature set, that `Map` is a wrapper over
`BTreeMap<String, Value>` (alphabetical key order, deterministic).
This means two requests with the same logical args but differing
key order on the wire produce the **same** digest after
canonicalization. That matches the (corrected) doc on digest.rs. ✓

Verified by reading Cargo.lock: `serde_json v1.0.149` is the only
serde_json in the tree. Default features active (no
`preserve_order`). ✓

### stats output determinism
stats.rs::compute_by_tool uses `BTreeMap<String, Vec<Span>>`
(stats.rs:93). Iteration is alphabetical, render_table walks
`by_tool` in iteration order (stats.rs:125). Same input → same
table. ✓

### JSONL store crash safety
store.rs:99-118 has no `BufWriter` — `JsonlStore` writes directly
to the `File` under a Mutex. Each `record()` is two `write_all`
calls (line + `\n`); on Linux/macOS those go straight to the page
cache without intermediate buffering. `flush()` calls
`File::flush()` which on POSIX is essentially a no-op for `File`
(it implements `Write::flush` as a no-op since there's no
in-process buffer). True durability requires `fsync`, which is
NOT performed.

This is acceptable for an observability tool: spans are best-effort
("data plane integrity > observation completeness") and the
exporter path is the durable trust boundary. README does not
promise crash-safe local persistence; the JSONL store is documented
as a convenience. Not a bug for v1, would be a v1.1 nice-to-have
behind an `--fsync` flag.

## Non-bugs investigated but rejected

- **Plain `+` instead of `saturating_add` in OtlpJsonExporter::render_span**
  (exporter.rs:237): for any realistic `start_unix_nanos` in the
  current epoch, `start_ns + u128::from(duration_ms) * 1_000_000`
  cannot approach `u128::MAX`. Span::end_unix_nanos uses saturating
  for the test `end_unix_nanos_saturates` extreme case; the OTLP path
  is not exercised by that scenario. Cosmetic inconsistency at most.
- **JSONL store no fsync**: documented "best-effort observability";
  v1.1 work item.
- **`as u64` truncation in `seed()`**: deliberately uses low 64 bits
  of nanos epoch as entropy mixed with monotonic counter. Not a bug.
- **No tokio::signal::ctrl_c() handler**: hard-kill on Ctrl+C is a
  documented v1 limitation; cycle D analyzed and accepted.
- **Empty `examples/` directory**: cosmetic; not blocking ship.
- **Zipkin `annotations: []` for error spans** (exporter.rs:155):
  empty array is a no-op for downstream tools but follows the Zipkin
  v2 schema (annotations is an array). Inert. Not a bug.
- **`response_status` returns `Some(SpanStatus::Ok)` for `is_response`
  with neither result nor error**: defensive default; finalize() uses
  it correctly. Not a bug.
- **`BufReader::lines()` strips trailing CR/LF and breaks on
  non-UTF-8**: JSON-RPC over stdio is UTF-8 by spec. Acceptable.
- **`SpanStatus::Timeout` variant unused at proxy runtime** (no
  per-request timeout in v1): D analyzed; load-bearing for v1.1.
- **`HttpArgs` deferred at runtime, not at parse time**: clap accepts
  it, main.rs returns a clear error. README documents it. Acceptable
  for v1.
