## Probes run in cycle D

### Cargo.toml dependency audit (NEW — neither A/B/C ran this)
Ran `cargo tree --depth 1` and grepped every `use` statement across
src/. Found: **`time = "0.3"` declared as a direct dependency, but
zero source files import it.** Only `std::time::{SystemTime, Duration,
Instant, UNIX_EPOCH}` is referenced (which is in core, not the `time`

### Shutdown handling — pending finalization
On EOF from child (proxy.rs:397-409), c2a task drains the pending
HashMap, finalizes each remaining entry with `response = None`
(producing `SpanStatus::Orphan`), and pushes to the observer
channel. Observer task drains until channel-close, then calls
`store.flush()`. Both `a2c` and `c2a` then `.await`, observer
`.await`, then `child.kill().await`. Pending requests **are** drained
on graceful EOF. ✓

### Shutdown — Ctrl+C / SIGINT (Windows)
The proxy does NOT install a `tokio::signal::ctrl_c()` handler.
On Ctrl+C, the runtime is killed by the OS without giving the
observer task a chance to drain. This is a v1 limitation but the
JSONL store flushes line-per-record (no buffer in our code beyond the
OS pipe), so written lines persist. Pending in-flight spans are
lost on hard-kill. README does not promise graceful Ctrl+C. Not a
correctness regression, documented as a v1 design choice in
.eval-notes-A. Not a bug to fix in this cycle.

### JSONL append atomicity across writes
`JsonlStore::record` (store.rs:100-110) does TWO `write_all` calls
(line bytes, then `b"\n"`) under one Mutex guard. Within a process
this is fully serialized. Across processes on Windows, NTFS
`FILE_APPEND_DATA` is OS-serialized; on POSIX, `O_APPEND` is atomic
only for writes ≤ PIPE_BUF. With two writes per record, two processes
appending could in theory interleave a record with the trailing LF of
another record. The module doc on store.rs:14-21 already documents
this as "single-writer per file is the supported deployment
model"; the proxy itself is single-writer. Acceptable v1 limitation.

A defensive fix would be to format the line+LF into one buffer and
issue one `write_all`. Worth doing in v1.1 if multi-process append
ever becomes a use case, but not a correctness bug today.

### Exporter batching
StdoutExporter, ZipkinExporter, OtlpJsonExporter all `export_batch`
synchronously per call. proxy.rs::observe (line 215) calls
`export_batch(std::slice::from_ref(span))` per single span — one
HTTP POST per span. This is intentionally simple and matches the
"data plane integrity > observation completeness" philosophy. Not a
bug for v1, but a perf opportunity for v1.1 (batching with timeout
flush). Out of scope for eval D.

### SLO window rolling
`evaluate` (slo.rs:260-319) computes
`cutoff = now_nanos.saturating_sub(slo.window_nanos)` and filters
spans with `start_unix_nanos >= cutoff`. The "now" for offline
`slo check` is the latest span timestamp (main.rs:170-176
`latest_or_now`). For real-time evaluation a caller would pass
`Span::now_unix_nanos()`. The window slides correctly. ✓

### SLO percentile across window
Lines 297-302: `ok_durs` is built from `relevant.iter()` (already
window-filtered) and only `SpanStatus::Ok` spans, then sorted, then
`percentile_sorted(&ok_durs, 0.95)`. Errors are excluded from the
latency budget — which is the right call (a 5xx that returns in 5ms
is not a "fast" success). ✓

## Bugs found

### BUG-3 (LOW, BUILD/SECURITY): unused `time = "0.3"` direct dependency

**Location**: `Cargo.toml` line 33.

**Evidence**:
- `Cargo.toml` declared `time = { version = "0.3", features = ["formatting", "parsing", "serde"] }` as a direct dep.
- `Grep "use time|::time::|extern crate time"` across `src/` returns zero matches outside `std::time::*`.
- `cargo tree -i time` shows `time v0.3.47` has only `mcptrace` as a reverse dependent — i.e. nothing else in the dep tree pulls it.
- `Cargo.lock` includes `time v0.3.47` and its transitive deps purely for the unused declaration.

**Impact**:
- Wasted compile time (a dozen extra crate compiles on cold builds).
- Larger attack surface for downstream `cargo audit` runs.
- More audit/license work for any consumer vendoring the dep tree.
- Bigger `Cargo.lock` diff churn on every dep refresh.
- For a v1.0 release of an observability/proxy tool that explicitly markets itself on minimal dependencies and a single-binary footprint (README "Why mcptrace" section), shipping a wholly unused crate in the lockfile is a real polish bug.

For a tool whose value prop is "tiny dependency footprint, transparent proxy, every byte audited" (`#![forbid(unsafe_code)]`, `#![deny(warnings)]`, `#![warn(clippy::pedantic)]`), an unused direct dep that's been sitting in `Cargo.toml` since the initial scaffold contradicts the contract.

**Fix**:
1. Delete the `time = ...` line from `[dependencies]` in `Cargo.toml`.
2. Re-run `cargo build --release` (clean) and `cargo test --release` (199/199 passing) to verify nothing was load-bearing.
3. Re-run `cargo tree -i time` → "did not match any packages" (confirmed gone).
4. Add a CHANGELOG.md entry under `[1.0.0]` Fixed.

No source-code changes required — the dep was never imported.

**Files touched**:
- `Cargo.toml` (1 line removed)
- `Cargo.lock` (auto-updated by cargo build)
- `CHANGELOG.md` (Fixed entry)

## Non-bugs investigated but rejected

- **Two `write_all` calls per JSONL record**: documented as
  single-writer-per-file deployment, Mutex serializes within process,
  Windows NTFS serializes across processes for FILE_APPEND_DATA.
  Acceptable for v1.
- **No `tokio::signal::ctrl_c()` handler**: hard-kill on Ctrl+C is a
  documented v1 limitation; JSONL store writes line-per-record so
  persisted data is intact. v1.1 work item.
- **Single-span exports (no batching)**: explicit design choice
  documented in exporter.rs module doc as "data-plane integrity >
  observation completeness". v1.1 perf opportunity.
- **`SpanStatus::Timeout` dead variant**: load-bearing for the v1.1
  per-request timeout story. Not removing.
- **Empty `examples/` directory**: minor cosmetic cleanup, not a bug.
  Could be removed in a future tidy-up; not blocking ship.
- **`#[allow(clippy::cast_*)]` crate-wide**: necessary for the
  u128↔u64↔f64 conversions that exporters and SLO math need; each
  cast site is bounded (saturating, try_from, or already validated).
- **Stats `--tool` is exact match, not glob**: documented as such,
  not a bug, README example uses exact name.
- **No retry on exporter failure**: explicit module-doc design choice.
