## Probes run in cycle C

### Concurrent-id collision (B flagged for later)

Constructed a unit-level probe driving the same code path the proxy
uses: two requests with the same JSON-RPC id, inserted into the same
`HashMap<String, Pending>`. Result: the second `insert` returns
`Some(_)` (the first Pending is silently dropped) and the first
request never finalizes — neither into a span nor into any counter.

For an observability tool whose tagline is "trace every tool call",
this is a silent observation drop with **no operator-visible signal**.
That's a bug for the contract this tool is selling. → Fixed in C
(see "Bugs found").

### JSON-RPC notification

`Frame::is_notification` returns true iff method is set and id is
absent. proxy.rs:299 only constructs a Pending for
`is_request() && !is_notification()`. Notifications are forwarded but
never get a span. Reasonable for a request/response correlation tool;
README's "tools/call" framing makes this implicit. NOT a bug.

### JSON-RPC batch (top-level JSON array)

`Frame::parse_with_limit` rejects non-object frames with
`Error::InvalidJsonRpc("frame is not a JSON object")` (jsonrpc.rs:43).
A batch (top-level array) is therefore parse-error'd, the byte stream
is still forwarded to the child (proxy.rs:317-321 "forwarding anyway"),
and the parse_errors counter increments. So mcptrace cannot extract
spans from batches, but it doesn't break the child either. README is
silent on batches; given MCP itself uses request-per-line in practice,
this is acceptable. Not a bug.

### JSON-RPC error response correlation

`Frame::is_response` returns true if `result` OR `error` is present
(jsonrpc.rs:84-87). `finalize` (proxy.rs:105-118) populates
`status=Error`, `error_code` from `error.code`, and
`error_message_digest = sha256(error.message)`. Verified by
`finalize_error_response_has_digest_not_message` test. The plaintext
message is **not** in the span. ✓

### Slow-child memory leak

There is no per-request timeout in v1.0 (`SpanStatus::Timeout` is
the dead variant B noted). A request that never gets a response sits
in the `pending` HashMap until child EOF, at which point the c2a
task drains the map and finalizes each entry as
`SpanStatus::Orphan`. Memory is bounded by in-flight request count.
For a misbehaving never-replying child, mcptrace will hold a Pending
forever — but so will the agent (it's blocked waiting too). Not a
mcptrace bug.

### Graceful shutdown / orphan finalization

c2a task on EOF drains pending → finalizes as Orphan → emits to
observer channel → flushes store via `observer_store.flush()`. ✓

### Exporter HTTP retry

ZipkinExporter and OtlpJsonExporter do NOT retry on failure. They
log the error via `Err(Error::Http(...))` which `observe()` writes
to stderr. Documented in module-doc: "If an export fails, the batch
is dropped and the failure is logged. This matches the observability
philosophy 'data-plane integrity > observation completeness'."
Not a bug.

### Zipkin v2 wire shape

- `id` (span_id): 16 hex chars from `new_span_id()` ✓
- `traceId`: 32 hex chars from `new_trace_id()` (Zipkin spec accepts
  16 OR 32 hex chars for traceId) ✓
- `timestamp`: micros (`s.start_unix_nanos / 1_000`), tested
- `duration`: micros (`u128::from(s.duration_ms) * 1_000`), tested
- `kind = "CLIENT"`, `localEndpoint.serviceName`, `tags.mcp.*` ✓

### OTLP/JSON wire shape

- `traceId`/`spanId`: hex strings. Per the OTLP/JSON spec section
  ("JSON Protobuf Encoding"), bytes fields like trace_id and span_id
  are explicitly encoded as **lowercase hex strings**, NOT base64
  (this is a documented override of proto3 JSON's base64 default).
  mcptrace produces lowercase hex via `format!("{:016x}")` /
  `format!("{:016x}{:016x}")`. ✓ Spec-conformant.
- `attributes[]` values use the AnyValue wrapper (`stringValue`,
  `intValue`) — verified by reading `render_span` lines 242-256. ✓
- `startTimeUnixNano` / `endTimeUnixNano` are stringified (proto3
  JSON requires uint64 as string) — verified by the test
  `otlp_start_end_time_strings`. ✓
- `kind: 3` = CLIENT, `status.code: 1|2` = OK|ERROR. ✓

### Span schema (README §"Span schema") parity

Walked every field in the README field-reference table against
`Span` struct (span.rs:62-89) and `Span::to_jsonl_line`. All 14
documented fields are present and match the documented type. ✓

### Stats sort stability

`compute_by_tool` returns `BTreeMap<String, GroupStats>`, which is
sorted-by-key. Render iterates in BTreeMap order. Output is stable
across runs. ✓

### SLO multi-tool filter

`SloConfigEntry::tool` is a single string. Multi-tool would require
a `Vec<String>` or comma-list. README example uses a single tool.
Minor limitation, not contradicted by docs. Not a bug.

### Rust panic surfaces

Re-grepped `unwrap()|expect(|panic!|unreachable!()` across `src/`.
- All test `unwrap()`s are in `#[cfg(test)]` modules.
- `unreachable!()` lives only in main.rs after CLI validation has
  already filtered the allowed set against the same allow-list the
  match arms cover. Truly unreachable.
- `finalize` uses `unwrap_or_else` fallback to never panic.
- `seed()` uses `map_or(0, ...)` for the SystemTime path.
- No raw `[i]` indexing on user-sized vecs. `windows(n)` not used.

### Arithmetic edge cases

- `duration.as_millis()` returns `u128`; cast to `u64` via
  `try_from(...).unwrap_or(u64::MAX)` (proxy.rs:104). Saturates,
  no panic.
- `Span::end_unix_nanos` uses `saturating_add` — covered by
  `end_unix_nanos_saturates` test.
- `parse_duration_nanos` uses `checked_mul` — covered by
  `parse_huge_overflows` test.

### trace_id collision probability

`splitmix64(SystemTime::now() ^ counter)` per call, twice per trace
id. Within one process the counter is monotonic so collisions
are negligible. Across two processes started in the same nanosecond,
the 128-bit space + counter mix makes a collision astronomically
unlikely (2^-64 per id pair even with identical seeds, since
splitmix64 is a bijection). Tested with the
`trace_ids_are_unique_across_1k_calls` test. Not a bug.

### JSONL store on disk full

`File::write_all` returns `io::Error` on ENOSPC. `JsonlStore::record`
propagates it via `?`. `proxy::observe` logs it via
`eprintln!("mcptrace: store.record error: {e}")`. The proxy data
path keeps running (transparency contract). Acceptable.

### JSONL store concurrent writers

`Mutex<File>` serializes writes within one process (correct).
Cross-process append on POSIX is atomic for ≤ PIPE_BUF; on Windows
`FILE_APPEND_DATA` is also serialized at the OS level. Module doc
explains the tradeoff honestly. Not a bug.

## Bugs found

### BUG-2 (LOW-MEDIUM, OBSERVABILITY): id collision is silently dropped

**Location**: `src/proxy.rs::run_stdio_proxy` agent→child task,
the `Pending` insert path.

**Impact**: When an agent re-uses a JSON-RPC id while the previous
request with that id is still in flight, the second `insert()`
silently overwrites the first `Pending` entry. The first request
never finalizes:
- no span in the JSONL store
- no span exported to Zipkin / OTLP / stdout
- no counter incremented (`spans_dropped_buffer_full`,
  `frames_too_large`, `parse_errors`, `orphan_spans` all stay 0)
- no log line

For an observability tool whose explicit value proposition is
"trace every tool call", a silent drop with no operator-visible
signal contradicts the contract. JSON-RPC id reuse is a protocol
violation per the spec, so the *forwarding* behavior (let the child
deal with it) is correct — but mcptrace should at minimum tell the
operator a drop happened.

This was flagged by eval-B as "out of scope, polish issue, not a
correctness one." I disagree: for a tool that exists to surface
silent observation drops, a silent observation drop *is* the
correctness contract.

**Fix**:

1. New counter on `ProxyMetrics`:
   `pub spans_dropped_id_collision: AtomicU64`.
2. After `insert(key, pend)` in the a2c task, check the returned
   `Option`; bump the counter if `Some(_)`.
3. Surface the counter in the proxy shutdown summary line printed
   by `main.rs::run_proxy_stdio`.
4. Regression test `pending_insert_collision_is_observable` in
   `proxy.rs::tests` exercises the insert path with two frames
   that share the same id_key, asserts the counter goes from 0
   → 1 and the map size stays at 1.
5. Default-state test `proxy_metrics_default_includes_id_collision`
   pins the new field at zero.

The forwarding behavior is unchanged. Both frames still reach the
child. Only the operator-visible counter is new.

**Files touched**:
- `src/proxy.rs` (counter + insert hook + 2 tests)
- `src/main.rs` (shutdown summary line)
- `CHANGELOG.md` (Fixed entry)
- `README.md` (test count badge: 197 → 199, tests section)

## Non-bugs investigated but rejected

- **`SpanStatus::Timeout` dead variant**: there is no per-request
  timeout in v1.0 so the variant is never produced. The match arm
  in `Zipkin::render_span` (line 154) and `OtlpJsonExporter`
  (line 240) is dead but reads cleanly and is load-bearing for the
  v1.1 streaming-mode story. Not removing.
- **JSON-RPC notifications never produce a span**: by design — they
  have no id and no response, so there's nothing to correlate.
- **Top-level JSON array (batch)**: parse-rejected, byte-forwarded,
  parse_errors counter bumps. README is silent but the "transparent
  proxy" contract is preserved.
- **Non-UTF-8 bytes from child**: `BufReader::lines()` on tokio
  stdio decodes as UTF-8 and would error on a non-UTF-8 line, ending
  that direction of the proxy. JSON-RPC over stdio is UTF-8 by
  spec; this matches what the MCP wire actually carries. Eval A
  documented this. Acceptable v1 constraint.
- **Exporter no-retry**: explicit module-level design comment
  "data-plane integrity > observation completeness". Not a bug.
- **Stats halts on bad JSONL line vs skipping**: strict-by-default
  is what an observability/forensic tool wants. Eval A test
  `read_jsonl_surfaces_bad_line` confirms the design.
