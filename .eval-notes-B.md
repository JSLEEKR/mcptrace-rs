## Findings

### Doc inconsistency (fixed in this cycle)

`src/digest.rs::arg_digest` doc claimed "we do **not** re-serialize — hashing the
canonical wire bytes". But the actual proxy path goes through
`Frame::arguments_canonical()` which calls `serde_json::to_vec(args)`, i.e. it
DOES re-serialize. The function-level behavior (hash whatever bytes you pass in)
is correct, but the doc misled readers about what the proxy actually digests.

Net effect of the real behavior: digests collapse semantically-equivalent
argument objects (different whitespace, different key order) to the same hash,
which is what operators want for repeat-call deduplication. Stability is tied
to serde_json's canonical form (BTreeMap-sorted keys, no whitespace) rather
than byte-exact wire input.

Fixed the doc to describe reality and call out the stability contract.

### Minor diagnostic gap (not a bug, noted)

`proxy.rs::a2c` task: if an agent re-uses a JSON-RPC id while one is still
in flight, the second `insert` silently overwrites the first `Pending` entry.
The first request never finalizes — neither into a span nor into the orphan
counter. JSON-RPC id reuse is an agent bug per spec, so this is acceptable
behavior, but a counter bump (`spans_dropped_id_collision` or similar) would
be useful operator signal. Not fixing in cycle B — out of scope, would
require a new metric field and is a polish issue, not a correctness one.

### Dead variant (not a bug, noted)

`SpanStatus::Timeout` exists in the enum but is never produced by the proxy
(no per-request timeout in v1.0). Documented as v1.1 streaming feature.
The match arm in `finalize` for `Some(SpanStatus::Timeout)` is dead code
but reads cleanly. Not removing — it's load-bearing for v1.1.
